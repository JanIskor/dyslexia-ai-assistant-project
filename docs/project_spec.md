# Техническое задание (Master Specification)
## Проект: ИИ-ассистент для персонализированной адаптации и подачи учебных текстов обучающимся с дислексией

---

## 1. Общая информация о проекте

### 1.1. Наименование системы
Интеллектуальная веб-система для персонализированной адаптации учебных текстов обучающимся с дислексией.

### 1.2. Назначение системы
Система предназначена для адаптации учебных материалов под особенности конкретного ученика с дислексией, передачи адаптированного текста ученику, проверки понимания материала и накопления данных для последующей персонализации.

### 1.3. Формат реализации
Система реализуется как веб-приложение с разделением на:
- frontend;
- backend;
- базу данных;
- модуль аутентификации и авторизации;
- модуль адаптации текста;
- RAG-подсистему;
- модуль чтения адаптированного текста;
- модуль тестирования;
- модуль обратной связи;
- модуль персонализации и обновления профиля ученика.

### 1.4. Статус проекта
Проект разрабатывается как **MVP для выпускной квалификационной работы** с перспективой дальнейшего расширения.

---

## 2. Цель проекта

Создать рабочий MVP-продукт, который позволяет:
- преподавателю авторизоваться в системе;
- выбрать ученика;
- ввести или загрузить учебный текст;
- автоматически адаптировать текст под конкретного ученика;
- отправить адаптированный текст ученику;
- дать ученику возможность прочитать материал в специальном интерфейсе;
- проверить понимание текста с помощью теста;
- получить обратную связь;
- использовать историю взаимодействий для последующих адаптаций.

---

## 3. Продуктовая гипотеза

**Персонализированная адаптация текста с использованием LLM, RAG и последующего тестирования понимания позволяет повысить читаемость и усвоение учебных материалов учащимися с дислексией по сравнению с универсальной, неперсонализированной адаптацией.**

---

## 4. Целевая предметная область

Система ориентирована на образовательный сценарий, в котором:
- преподаватель подбирает или вводит учебный материал;
- система адаптирует текст под конкретного ученика;
- ученик получает адаптированную версию текста;
- ученик читает материал;
- после чтения ученик проходит тест;
- система анализирует результаты и улучшает последующие адаптации.

## 5. Основные роли пользователей

### 5.1. Преподаватель
Основная роль MVP.

#### Возможности преподавателя:
- регистрация и вход в систему;
- просмотр своего кабинета;
- создание и редактирование карточек учеников;
- просмотр списка учеников;
- ввод текста для адаптации;
- выбор ученика, под которого будет выполняться адаптация;
- запуск адаптации;
- просмотр результата адаптации;
- отправка адаптированного текста одному ученику;
- отправка адаптированного текста нескольким ученикам;
- просмотр истории материалов;
- просмотр результатов тестирования и обратной связи учеников.

### 5.2. Ученик
Конечный получатель адаптированного материала.

#### Возможности ученика:
- регистрация и вход в систему;
- просмотр присланных материалов;
- чтение адаптированного текста;
- использование dyslexia-friendly интерфейса;
- прохождение теста после чтения;
- отправка обратной связи;
- просмотр своей истории материалов и результатов.

### 5.3. Администратор
Роль предусматривается архитектурно, но не является обязательной для MVP.

#### Возможные функции администратора:
- управление пользователями;
- просмотр системных логов;
- управление глобальными настройками;
- контроль корректности работы системы.

---

## 6. Основной пользовательский сценарий

### 6.1. Сценарий преподавателя
1. Преподаватель входит в систему.
2. Преподаватель выбирает ученика из списка.
3. Преподаватель вводит учебный текст.
4. Система анализирует профиль ученика и связанные данные.
5. Система автоматически выбирает стратегию адаптации.
6. Система генерирует адаптированную версию текста.
7. Преподаватель просматривает результат.
8. Преподаватель отправляет материал выбранному ученику.
9. При необходимости преподаватель отправляет тот же материал другим ученикам.

### 6.2. Сценарий ученика
1. Ученик входит в систему.
2. Ученик видит присланный материал.
3. Ученик открывает и читает адаптированный текст.
4. После чтения ученик проходит тест.
5. Ученик оставляет обратную связь.
6. Система сохраняет результаты.

### 6.3. Сценарий последующей адаптации
1. Преподаватель снова выбирает ученика.
2. Вводит новый текст.
3. Система использует:
   - профиль ученика;
   - историю взаимодействий;
   - результаты предыдущих адаптаций;
   - ответы на тесты;
   - обратную связь;
   - правила адаптации.
4. Система формирует новую, более персонализированную адаптацию.

## 7. Принципы проектирования системы

### 7.1. Преподаватель не является экспертом по дислексии
Преподаватель не должен вручную принимать экспертные решения по типу адаптации текста.  
Выбор адаптационной стратегии в MVP должен выполняться системой автоматически на основе правил и данных о пользователе.

### 7.2. Персонализация не через дообучение модели
В первой версии персонализация реализуется **не через fine-tuning LLM**, а через:
- профиль ученика;
- retrieval персонального контекста;
- retrieval правил адаптации;
- динамическую сборку prompt;
- rule-based выбор стратегии;
- историю результатов.

### 7.3. Пошаговая разработка
Проект реализуется поэтапно, небольшими end-to-end блоками.  
Не допускается попытка сгенерировать всю систему одним промптом.

### 7.4. Стабильность важнее скорости генерации
Каждый шаг должен:
- быть ограниченным по объёму;
- не ломать существующий код;
- сопровождаться обновлением документации;
- завершаться коммитом в git.

## 8. Функциональные требования

### 8.1. Аутентификация и авторизация
Система должна обеспечивать:
- регистрацию пользователя;
- вход пользователя;
- хранение ролей;
- разграничение доступа по ролям;
- защищённые маршруты.

### 8.2. Работа с профилями пользователей
Система должна обеспечивать:
- создание карточки ученика;
- редактирование карточки ученика;
- связь преподавателя с учениками;
- хранение параметров адаптации.

### 8.3. Работа с материалами
Система должна обеспечивать:
- ввод текста преподавателем через интерфейс;
- сохранение материала;
- выбор языка текста;
- привязку материала к ученику или группе учеников.

### 8.4. Модуль адаптации
Система должна:
- принимать исходный текст;
- получать профиль ученика;
- учитывать историю ученика;
- извлекать правила адаптации;
- вызывать LLM;
- сохранять результат адаптации.

### 8.5. Модуль отправки материалов
Система должна:
- отправлять адаптированный материал одному ученику;
- отправлять адаптированный материал нескольким ученикам;
- фиксировать факт отправки.

### 8.6. Модуль чтения
Система должна:
- отображать адаптированный текст;
- поддерживать специальный интерфейс чтения;
- давать доступ к пояснениям и структуре текста.

### 8.7. Модуль тестирования
Система должна:
- генерировать тест по адаптированному тексту;
- принимать ответы;
- вычислять score;
- сохранять результаты.

### 8.8. Модуль обратной связи
Система должна:
- принимать субъективную оценку понятности текста;
- принимать оценку удобства чтения;
- принимать текстовый комментарий;
- сохранять обратную связь.

### 8.9. Модуль персонализации
Система должна:
- обновлять агрегированное представление о пользователе;
- хранить успешные и неуспешные параметры адаптации;
- использовать результаты прошлых сессий при следующих адаптациях.

## 9. Концептуальная логика адаптации

На текущем этапе фиксируется, что система должна поддерживать несколько стратегий адаптации, а их окончательный состав может уточняться в исследовательской части.

### 9.1. Strategy A — Simplification + Visual Support
- упрощение лексики;
- сокращение длины предложений;
- упрощение структуры текста;
- визуальная сегментация.

### 9.2. Strategy B — Markup Only
- исходный текст в основном сохраняется;
- изменяется подача;
- добавляется визуальное выделение и деление на блоки.

### 9.3. Strategy C — Guided Reading
- частичное упрощение;
- деление текста на фрагменты;
- пояснения к сложным местам;
- усиленная структурность.

### 9.4. Strategy D — Vocabulary Support
- сложные слова остаются;
- текст сопровождается пояснениями и словарной поддержкой.

### 9.5. Strategy E — Visual Enrichment
- допускается добавление простых визуальных опор;
- в MVP не является обязательной функцией.

## 11. RAG-подсистема

В проекте должна использоваться RAG-архитектура, разделённая на три независимых источника.

### 11.1. RAG по учебному контенту
Назначение:
- поддержка учебной точности;
- извлечение терминов и определений;
- сохранение смысловой корректности.

Статус для MVP:
- архитектурно предусмотреть;
- активную реализацию можно отложить.

### 11.2. RAG по профилю пользователя
Назначение:
- извлечение данных о конкретном ученике;
- извлечение истории прошлых адаптаций;
- учёт результатов тестов и обратной связи.

Статус для MVP:
- обязательный элемент.

### 11.3. RAG по правилам адаптации
Назначение:
- хранение правил dyslexia-friendly presentation;
- хранение языковых правил;
- хранение системных инструкций и шаблонов.

Статус для MVP:
- обязательный элемент.

## 14. Dyslexia-friendly UI

Интерфейс чтения для ученика должен поддерживать:
- увеличенный межстрочный интервал;
- увеличенные интервалы между словами и буквами;
- ограниченную ширину строки;
- мягкие цветовые темы;
- высокий контраст при необходимости;
- деление текста на короткие блоки;
- выделение ключевых слов и фраз;
- возможность переключения режима отображения.

В MVP:
- реализуется упрощённая, но реальная версия такого интерфейса.

## 18. Границы MVP

### В MVP обязательно реализовать:
- регистрацию и вход;
- роли teacher и student;
- кабинет преподавателя;
- кабинет ученика;
- карточки учеников;
- ручной ввод текста;
- выбор ученика для адаптации;
- базовую адаптацию текста через LLM;
- retrieval по профилю ученика;
- retrieval по правилам адаптации;
- отправку адаптированного текста;
- чтение адаптированного текста;
- тест после чтения;
- обратную связь;
- обновление профиля ученика по итогам сессии.

### В MVP не обязательно реализовывать:
- загрузку PDF/DOCX;
- OCR;
- мобильное приложение;
- генерацию изображений для каждого фрагмента;
- голосовые функции;
- полноценный RAG по большому учебному корпусу;
- fine-tuning модели;
- сложную аналитическую панель;
- расширенный admin-функционал.

## 19. Принципы prompt engineering для разработки проекта

### 19.1. Нельзя писать один большой промпт на весь проект
Проект должен разрабатываться **маленькими шагами**.

### 19.2. Каждый промпт должен:
- учитывать уже существующий код;
- не ломать структуру проекта;
- иметь ограниченный scope;
- описывать, что входит и что не входит в задачу;
- требовать обновления docs;
- предусматривать будущее расширение.

### 19.3. Каждый новый шаг должен сопровождаться:
- отдельной веткой git;
- коммитом;
- обновлением документации;
- проверкой ручных сценариев.

---

## 20. Правильная стратегия разработки

### 20.1. Разработка должна идти не “всё сразу”, а маленькими end-to-end шагами
Правильный пример:
1. Инициализация проекта;
2. Регистрация/вход + локальная БД + роли;
3. Ролевые кабинеты-заглушки;
4. CRUD учеников;
5. Ввод и сохранение материалов;
6. Базовая адаптация;
7. Отправка материалов;
8. Reader UI;
9. Тестирование;
10. Feedback;
11. Обновление профиля;
12. Улучшение retrieval и персонализации.

### 20.2. Предпочтительный принцип
Лучше делать:
- маленький, законченный кусок от интерфейса до backend;
чем
- сначала весь frontend без реальной логики,
- а потом пытаться под него догонять backend.

## 21. Рекомендуемая стратегия работы с git

### 21.1. Основной принцип
Каждый блок реализуется в отдельной ветке.

## 23. Текущий согласованный подход к разработке

На текущем этапе согласовано следующее:

1. Проект строится как реальная ролевая веб-система.
2. Основной пользователь MVP — преподаватель.
3. Конечный пользователь адаптированного материала — ученик.
4. UI MVP должен быть русскоязычным.
5. Разработка идёт по шагам.
6. Codex используется как инструмент реализации отдельных блоков.
7. Каждый шаг должен быть маленьким и контролируемым.
8. Документация должна жить внутри проекта и обновляться вместе с кодом.
9. В начале допустимо использовать локальную БД, если это упрощает разработку и проверку сценариев.
10. Архитектура должна сразу предусматривать дальнейшее расширение до полноценного продукта.

## 24. Что считается ошибочным подходом

Неправильно:
- пытаться сразу сгенерировать весь проект;
- реализовывать большие несвязанные куски;
- не фиксировать изменения в git;
- не обновлять docs;
- хардкодить логику без учёта расширения;
- заставлять преподавателя вручную принимать экспертные решения по адаптации;
- делать UI без реального backend-сценария, если следующий шаг должен быть end-to-end.

---

## 25. Что считается правильным подходом

Правильно:
- двигаться маленькими шагами;
- сначала проектировать и фиксировать ограничения;
- после каждого шага проверять сценарии;
- коммитить после каждого стабильного блока;
- вести master-spec;
- использовать Codex итеративно;
- проверять то, что генерирует Codex, а не принимать всё без анализа;
- строить устойчивую архитектуру, а не временные костыли.

## 26. Критерии успешности проекта в целом

Проект можно считать успешным, если:
- реализован рабочий MVP;
- есть реальный end-to-end сценарий teacher → student;
- есть адаптация текста;
- есть чтение в специальном интерфейсе;
- есть тестирование и обратная связь;
- есть базовая персонализация;
- архитектура не разваливается при добавлении новых блоков;
- система пригодна для демонстрации и описания в ВКР.

Проект разрабатывается как последовательность небольших, контролируемых, документируемых шагов, где каждый шаг:
